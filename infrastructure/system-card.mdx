## Architecture Overview

HANA is not a monolithic LLM wrapper. It is a **dual-model architecture** that separates clinical reasoning from real-time conversation — solving the fundamental tension between intelligence and latency in healthcare AI.

## Design Principles

### 1. Separation of Reasoning and Conversation

The most critical architectural decision in HANA is the **decoupling of the reasoning model from the conversational model**. This solves a fundamental problem: smart reasoning requires time, but patient conversations require low latency.

| Layer | Function | Latency Requirement |
|-------|----------|---------------------|
| **Reasoning Engine** | Analyze data, determine questions, apply protocols | Async (pre-call, seconds to minutes) |
| **Conversational Agent** | Execute conversations with patients | Real-time (<500ms response) |

The Reasoning Engine pre-computes the optimal set of questions before each interaction. The Conversational Agent then delivers these questions naturally, without needing to reason in real-time — eliminating the latency penalty of complex reasoning.

### 2. Low User Burden

The system is designed around the principle that **the patient should do no work**. This means:

- No app downloads or logins
- No behavior changes required
- No redundant questions (the system reads existing data first)
- Channel selection adapts to patient preference (voice vs. text, time of day)

### 3. Protocol-First, Not Model-First

HANA does not invent clinical protocols. It **executes existing protocols** that clinics have already validated. The AI's job is delivery optimization — getting the right protocol to the right patient at the right time through the right channel.

### 4. Infrastructure, Not Application

HANA is designed as a composable infrastructure layer, not a standalone application. This means:

- Clinics can use it directly (HANA Direct)
- Platforms can build on top of it (HANA Connect / SDK)
- Outputs feed into existing workflows, not replace them

### Interaction Scoring

HANA does not measure success by conversation count or minutes. Every interaction is scored across multiple dimensions:

| Dimension | What It Measures |
|-----------|------------------|
| **Relevancy** | Did the question elicit clinically useful information? |
| **Safety** | Were risk signals properly detected and routed? |
| **Completeness** | Were all protocol-required data points captured? |
| **Patient Burden** | Was the interaction appropriately brief and non-redundant? |
| **Channel Fit** | Was the right channel used for this patient and context? |

<Info>
  HANA calculates **interactions**, not conversations or minutes. A single conversation may contain 15 interactions. The quality of each interaction is what drives clinical value.
</Info>

## Model Stack

### Current Models

| Component | Model | Hosting |
|-----------|-------|---------|
| **Reasoning Engine** | OSS (Open Source) fine-tuned model | Self-hosted (EU servers) |
| **Voice Agent** | LLaMA 3.1 (fine-tuned) | Self-hosted |
| **Risk Detection** | Custom NLP classifiers | Self-hosted |
| **Speech-to-Text** | Configurable (Deepgram, Whisper) | Self-hosted / Cloud |
| **Text-to-Speech** | Configurable (voice selection per deployment) | Self-hosted / Cloud |

### Why Self-Hosted

Six months ago, HANA migrated away from OpenAI to self-hosted open-source models. The reasons:

1. **Data Sovereignty** — Patient data never leaves controlled infrastructure
2. **GDPR Compliance** — EU data residency requirements satisfied at the infrastructure level
3. **On-Premise Capability** — Self-hosted models enable on-premise deployment for hospitals running their own servers
4. **Cost Control** — No per-token API costs at scale
5. **Model Customization** — Fine-tuning on 1M+ healthcare interactions creates a proprietary data moat

### Data Flywheel

```
More Patients → More Interactions → Better Reasoning Model → Better Outcomes → More Clinics → More Patients
```

The reasoning model improves with every interaction. This creates a compounding advantage that cannot be replicated by competitors using off-the-shelf foundation models.

## Input Layers

The Reasoning Engine ingests data from five distinct input layers:

### 1. EHR & Patient Data
- Demographics, medical history, active medications
- Lab results, vitals, care plans
- Pulled via direct API integration or EHR connectors (Redox, Catagon)

### 2. Memory Layer
- Past conversation history and outcomes
- Patient interaction preferences (channel, time, complexity)
- Longitudinal patterns (improving, declining, stable)

### 3. Clinical Protocols
- Clinic-defined care protocols (PDF, Docs, structured knowledge bases)
- Standardized screening instruments (PHQ-9, GAD-7, DIVA, etc.)
- Escalation procedures and safety policies

### 4. Engagement & Personalization
- Optimal channel per patient (voice vs. text)
- Best time of day for engagement
- Language and cultural context
- Communication style (direct vs. conversational, simple vs. technical)

### 5. Safety & Observability
- Real-time risk signal detection
- Interaction scoring and quality metrics
- Compliance audit trail

## Limitations and Known Constraints

| Constraint | Details |
|------------|---------|
| **Not a diagnostic tool** | HANA gathers data and delivers protocols; it does not diagnose or prescribe |
| **Not a crisis hotline** | Risk detection triggers escalation to humans, not AI counseling |
| **EHR dependency** | Full bidirectional integration requires API access; some EHRs (e.g., eClinicalWorks) do not provide this |
| **Connectivity required** | Voice and SMS require phone network; WhatsApp requires internet |
| **Sequential evaluation** | Current clinical study used sequential (not randomized) design; independent RCTs are planned |
